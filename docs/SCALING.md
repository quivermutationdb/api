# Scaling audit: n ≤ 10, height ≤ 10

> Status: **planning only — nothing here is implemented.** This is the checklist
> for the phase-2 expansion (see memory `phase2-scaling`). It records every place
> the current code would break or silently misbehave when `max_vertices` goes from
> 4 → 10 and the edge-weight bound goes from 2 → 10, plus the soundness traps to
> avoid. Read this before touching generation.

## 0. The one conceptual thing to settle first

"Rank 10" and "height 10" scale **different** parts of the catalog, and conflating
them is the main risk:

- **Finite cluster type (A, D, E) and affine/Euclidean type (Ã, D̃, Ẽ)** are all
  simply-laced ⇒ every arrow has weight 1, except a handful of weight-2 arrows
  (e.g. Kronecker = affine A₁). They live at **height ≤ 2**. Raising the height
  bound adds *nothing* to these families.
- **Height ≥ 3 ⇒ mutation-infinite for rank ≥ 3** (Derksen–Owen: if any quiver in
  the class has an entry with |b_ij| ≥ 3 and rank ≥ 3, the class is of infinite
  mutation type). So a "height 10" *finite* class can only exist at **rank 2**
  (the (a,b) quivers with a·b ≤ 4). Every rank-≥3 quiver we add by raising the
  height lands in an **open/infinite** class that we can only partially explore.

Consequence: there is no single "exhaustive enumeration of all quivers with
n ≤ 10, weight ≤ 10". That set is (a) combinatorially impossible to enumerate (see
§1) and (b) almost entirely infinite-type. The expansion must be split into two
clearly separate jobs:

1. **Targeted finite/affine families** — directly construct A/D/E + Ã/D̃/Ẽ seeds
   for ranks ≤ 10, BFS each to completion, store as closed classes. Bounded,
   well-defined, the high-value content.
2. **Bounded-height census** — whatever "all quivers up to weight h" cut we still
   want for small n, generated by *orderly generation* (not the current brute
   force), and stored mostly as open classes with partial exploration.

The current `generate_seed_quivers` → BFS → glue pipeline conflates these. It only
works because at n=4/bound=2 the brute force is tiny.

## 1. Hard blockers (today's code will not run at n=10 at all)

### 1a. `core.generate_seed_quivers` — brute-force matrix enumeration
Iterates `product(range(-bound, bound+1), repeat=n(n-1)/2)`:

| n  | upper entries | bound=2 | bound=10 |
|----|---------------|---------|----------|
| 4  | 6   | 1.5e4   | 1e8   |
| 6  | 15  | 1e10    | 1e20  |
| 8  | 28  | 1e20    | 1e37  |
| 10 | 45  | **1e31**| **1e59** |

Infeasible even at bound=2, n=8. **Must be replaced**, not tuned. Options:
- For job (1): skip enumeration entirely — hand-construct the family seeds.
- For job (2): orderly generation / canonical augmentation (McKay) — grow quivers
  vertex-by-vertex, keeping one nauty-canonical representative per iso class,
  pruning with the height bound and (for rank ≥ 3) the Derksen–Owen ≥3 cutoff.
  Note even this produces astronomically many iso classes at n=10; a census is
  only storable for small n or with a low height cap.

### 1b. `canonicalize.PermutationCanonicalizer` — O(n!·n²) fallback
10! = 3.6M perms per call, and `canonical_form` is called constantly (every BFS
node, every quiver_id). **nauty becomes mandatory.** Today pynauty is optional with
a silent permutation fallback. At n=10 the fallback is unusable, so:
- Pin `pynauty` in `requirements.txt`.
- Make the backend non-silent at scale: fail loudly (or refuse to populate) if
  `active_backend() != "nauty"` when `max_vertices > 6`.
- Verify pynauty actually builds on Render (it has a C extension / nauty dep).

### 1c. `core.canonical_class_rep` — O(class_size · n!)
`min(_apply_permutation(m, perm) for m in labeled for perm in permutations(n))`.
At n=10 that's 3.6M perms × (10⁴–10⁵ members) per class. Catastrophic, and it runs
once per class in `_merge_orbits`. **Rewrite** to use nauty: the lex-min over all
relabelings of all members equals `min(canonical_form(m) for m in members)` because
nauty's `canonical_form` is already iso-invariant. One nauty call per member, then a
lex-min — drops n! entirely.

### 1d. `invariants.symmetry_group` — O(n!) automorphism search
Enumerates all n! permutations checking `b_{σ(i)σ(j)} == b_{ij}`. The docstring even
says "switch to nauty when scaling." **Replace** with `pynauty.autgrp` (returns
generators + group order directly). Keep `_group_name` as a best-effort label; at
n=10 most groups won't match the small hard-coded cases and will fall back to
"order N" — acceptable, but note it.

## 2. Soundness traps when raising the height bound

The invariant-correctness principle (memory `invariant-correctness-principle`):
never emit a wrong value; truncated search ⇒ *unknown*, never a definite answer.

### 2a. No node cap on BFS — runaway memory on infinite classes
`_bfs_orbit` has **no size/time budget**; it loops until the queue empties or a
mutation exceeds `bound`. Today an infinite class hits the weight-2 wall almost
immediately. With bound=10 an infinite class must reach |b| ≥ 11 before any single
mutation is flagged — so it explores *vastly* more matrices first and can OOM the
process before ever setting `is_open`.

Fix requires a **node/time cap with a sound third state**:
- `is_open` currently has two origins (closed = queue drained; open = weight wall).
  A cap introduces a third: **truncated**. A truncated class must NOT be stored as
  closed/finite. It must not get an exact `class_size`, must not get
  `is_mutation_acyclic = False`, must not get a Dynkin label.
- Cleanest: keep the **Derksen–Owen ≥3 cutoff for rank ≥ 3 regardless of `bound`**
  (reaching |b| ≥ 3 already proves infinite — no reason to keep exploring to 10),
  and only use a high `bound` for the rank-2 census where ≥3 does *not* imply
  infinite. This keeps infinite-class detection cheap and makes the high bound a
  rank-2-only concern.

### 2b. `is_open` semantics are binary in the schema
`MutationClass.is_open` is a bool, and lots of code maps `is_open → class_size None`,
`not is_open → finite`. A "truncated/unknown finiteness" state has no representation.
The finiteness trichotomy columns exist (`is_finite_confirmed`,
`is_infinite_confirmed`, `is_infinite_expected`) but `is_open` is still the load-
bearing flag in `_class_size`, `class_is_mutation_acyclic`, `dynkin.classify` gating,
and `_la_bounds`. Plumb a genuine three-state finiteness before raising the bound.

### 2c. Mutation-acyclic subquiver fallback depends on catalog completeness
`crud._resolve_mutation_acyclic` / `_has_known_non_ma_subquiver` upgrade
unknown → False when a member has an induced (n−1)-subquiver already known
not-mutation-acyclic, processed in ascending rank order. This is sound **only if the
lower-rank subquivers are present in the same generation run.** With *targeted*
seeding (job 1) we may seed E8 without seeding every rank-7 subquiver it contains, so
the fallback would silently find nothing and leave classes at unknown (still sound —
just lower coverage). If we want the propagation to keep working, the run must
include the closure under vertex deletion, or seed a known not-mutation-acyclic base
set (Markov, etc.) explicitly. Re-verify `Q.n4.128d2df494351df3` stays False.

## 3. Per-component scaling notes

### 3a. `dynkin.py` — reference table
- Add `_E6/_E7/_E8` builders and extend the A/D ranges to 10 in `_build_reference`
  (currently `max_rank=4`, A-only+D≥4). E is not a family — just the three.
- The reference is built by BFS-ing each seed (`explore_mutation_class(seed).mc_id`).
  Largest finite labeled orbits up to rank 10: **D₁₀ ≈ 136k**, A₁₀ ≈ 58.8k,
  E₈ = 25,080 seeds. Building the reference therefore runs several large BFS — do it
  **once** and persist/memoize (don't rebuild per process start). Consider caching
  the reference `{mc_id: name}` in the DB or a checked-in JSON.
- Affine/Euclidean: `classify` returns None for them (they're not finite Dynkin), so
  they'd be stored with `dynkin_type = NULL`. If we want Ã/D̃/Ẽ *labeled*, add an
  affine reference + a separate `affine_type` column (don't overload `dynkin_type`,
  which is documented as finite-only).

### 3b. `local_acyclicity.py` — Banff/Louise/P′ budgets
`_la_bounds`: closed → depth 64, timeout 15s; open → depth 8, timeout 3s. The
source-deletion recursion is roughly exponential in rank (it recurses on Q∖{k},
Q∖{j}, Q∖{k,j} and BFS-searches each). At rank 8–10 over classes with 10⁴–10⁵
members, most calls will **time out → "unknown"**. That's *sound* (unknown, not
wrong) but coverage drops sharply, and 15s × (one Banff + Louise + P′) × thousands of
classes is a long populate. Expect to (a) raise timeouts and run populate offline /
in batches, and (b) accept that high-rank local-acyclicity is mostly unknown. The
`cap=8` arrow cap is fine for weight-≤2 finite classes.

### 3c. `invariants.py`
- `representation_type` PSD test iterates `combinations(range(n), size)` over all
  2ⁿ principal submatrices — n=10 → 1024 dets, trivial. **Fine** to ~n=20. Tame/
  Euclidean detection via the Tits form works as-is. Good news for the affine push.
- `is_planar` returns None for n > 4 (so unknown for every rank ≥ 5). If planarity
  is wanted at scale, wire in a real test (e.g. `networkx.check_planarity`); else
  document it as deliberately unknown.
- `_group_name` only names specific small orders — see §1d.

### 3d. Storage / DB — the sleeper problem
`MutationClass.labeled_quivers` stores the **entire labeled orbit** as one JSONB
blob. For D₁₀ that's ~136k matrices × 100 ints ≈ tens of MB **in a single row**.
That will blow past sane row sizes, make `get_class_detail` huge, and the
`labelings`-scope expansion in `list_quivers`/`export_rows` loads it all into Python.
Before rank ~7–8 this needs rethinking:
- Store **distinct quivers + labeling_count** (already have the count on `Quiver`),
  not every labeling; reconstruct labelings on demand only if ever needed.
- Or move labeled matrices to a child table (`labeled_quiver(mc_id, qmd_id, matrix)`)
  with pagination, instead of a JSONB array.
- `boundary_quivers` JSONB has the same issue for big open classes.

Indexes: only `quivers.mc_id` is indexed today. At scale add indexes on the columns
the API filters/sorts on — `quivers.n_vertices`, `max_edge`, `is_connected`,
`is_acyclic`, `representation_type`, and `mutation_classes.n_vertices`,
`labeled_size`, `is_open`, the three-state property columns. Without them every
Browse/Search page is a full scan.

### 3e. Export path — unbounded materialization
`export_rows` does `query....all()` with **no pagination** and builds the full list
in memory, and the `labelings` scope multiplies rows by class size. A "download
everything" at n=10 could be millions of rows → OOM / request timeout on Render.
Needs streaming (server-side cursor / chunked CSV) and likely an async/offline export
for the big cuts. `exporters.to_xlsx_bytes` (openpyxl) is especially memory-heavy —
cap XLSX row count and steer large cuts to CSV.

### 3f. Gluing phase memory
`run_generation` holds **all** raw orbits in memory at once plus the `qid_to_orbits`
inverted index before merging. Fine for targeted seeding (hundreds of classes);
a problem only if job (2) census ever gets large — in which case gluing needs to
stream/checkpoint.

## 4. Suggested go-forward shape (when we do it)

1. Make nauty mandatory (§1b) and rewrite the three n! hotspots to use it
   (`canonical_class_rep`, `symmetry_group`, guarantee no permutation fallback).
2. Add a **targeted seed module**: explicit A/D/E + Ã/D̃/Ẽ constructors for ranks
   ≤ 10; BFS each to completion (these are weight ≤ 2, so the existing bound=2 BFS
   is correct — no height change needed for this job).
3. Add a **sound truncation state** (node/time cap → unknown finiteness) and keep the
   Derksen–Owen ≥3 cutoff for rank ≥ 3 so infinite classes stay cheap (§2a/2b).
4. Move large labeled orbits out of the single JSONB blob (§3d); add filter/sort
   indexes; stream exports (§3e).
5. Extend the Dynkin reference (E6/7/8, A/D to 10) and persist it; decide on an
   `affine_type` column (§3a).
6. Treat "height 10" as a **rank-2-only + small-n census** concern, not a global
   enumeration (§0); pick an explicit, storable scope for job (2).
7. Re-run the soundness regression checks (Markov base case,
   `Q.n4.128d2df494351df3` stays False) after the generation rework.

### What does NOT need work
- Tits-form representation type (finite/tame/wild) scales fine — the affine/Euclidean
  detection we want is already correct to well past rank 10.
- The three-state provenance design and the mutation-acyclic heredity argument are
  sound; they just need the catalog-completeness caveat in §2c respected.
